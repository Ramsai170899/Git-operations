The Big Picture

Despite all the terminology, Git fundamentally has only two primitives. Everything else is just control over which one happens and how.

‚ë† Move a pointer
Fast-forward ‚Äî no new commit needed, the branch label just slides forward.
‚ë° Create a merge commit
New commit with two (or more) parents ‚Äî preserves full history.
Core Merge Types

These define what kind of merge happens ‚Äî how your branch history looks after the operation.

‚ö°
Fast-Forward Merge
git merge feature
üìö Analogy: You're reading a book at chapter 5. Your friend reads ahead to chapter 10. To "merge" their progress ‚Äî you just flip your bookmark to chapter 10. No new chapter is created.

Happens when the target branch has no new commits since the feature branch diverged. Git simply moves the branch pointer forward ‚Äî no merge commit is created, history stays linear.

$git checkout main
$git merge feature # fast-forward by default
‚úì Clean linear history ‚úì No divergence ‚ö† Loses branch context
üîÄ
Three-Way (True) Merge
git merge feature
üìö Analogy: You and a colleague both edit different chapters of the same document independently. A merge commit is like a new title page that says "this combines both your work" ‚Äî and it has two authors.

Happens when both branches have diverged. Git finds the common ancestor (the "base"), compares changes from both branches, and creates a new merge commit with two parents.

$git checkout main && git merge feature
# Git finds common ancestor A, diffs both sides, creates M
‚úì Preserves full history ‚úì Shows branch context ‚ö† Non-linear graph
üè∑Ô∏è
No Fast-Forward Merge
git merge --no-ff feature
üìö Analogy: Even if your colleague only read ahead, you still create a meeting note that says "we discussed this at 3pm on Tuesday." It's a deliberate record that a decision point happened.

Forces a merge commit even when a fast-forward would be possible. Preserves the information that a branch existed. Great for feature branches where you want the history to show "this was a cohesive feature."

$git merge --no-ff feature # always creates M
‚úì Feature grouping visible ‚úì Easy to revert whole feature ‚Üí Team standard practice
üóúÔ∏è
Squash Merge
git merge --squash feature
üìö Analogy: You spend a week drafting a 20-page report with many messy drafts. Before handing it in, you compress it into 1 clean final page. Only the polished result ends up in the archive ‚Äî not every draft.

Collapses all commits from the feature branch into one staged change on the current branch. You then commit it manually. Keeps history clean, but loses individual commit history and branch relationship.

$git merge --squash feature
$git commit -m "feat: add user auth" # you must commit!
‚úì Clean main branch log ‚úó Loses individual commits ‚Üí PR merge on GitHub
üêô
Octopus Merge
git merge feat-a feat-b feat-c
üìö Analogy: A conductor cues all orchestra sections ‚Äî strings, brass, woodwinds ‚Äî simultaneously into one unified performance. One cue, many sources.

Merges more than two branches in a single merge commit. The resulting commit has multiple parents. Used primarily in large monorepos or integration branches. Requires all merges to be conflict-free.

$git merge feat-a feat-b feat-c
# Creates 1 commit with 4 parents (base + 3 branches)
‚úì Multiple independent features ‚úó Fails on conflicts ‚Üí Rare, integration branches
üí•

üìö Analogy: You and a colleague both edit the same sentence in the same paragraph. Git can't auto-pick one ‚Äî it puts both versions in the file and asks you to choose.

When both branches modify the same lines differently, Git can't auto-resolve. It marks conflict sections in the file and pauses. You edit manually, git add, then git merge --continue.

Auto-Merge (No Conflicts)
git merge feature ‚Üí Auto-merging...
üìö Analogy: You edit chapter 3, your colleague edits chapter 7. When combined, there's zero overlap ‚Äî Git merges them automatically without asking anyone.

When changes from both branches touch different parts of the codebase, Git can automatically combine them. It creates the merge commit without human input. This is the happy path.

$git merge feature
Auto-merging app.js
Auto-merging styles.css
Merge made by the 'ort' strategy.
‚úì No action required ‚úì Changes in different files/lines
Merge Strategies

These control how Git internally performs the merge algorithm. Most users never change these, but understanding them helps debug tricky situations.

üöÄ
ORT Strategy
git merge -s ort feature
üìö Analogy: The newest, fastest postal sorting algorithm. It handles all the edge cases better than its predecessor and is faster to boot. You don't see it working ‚Äî you just get better results.

Ostensibly Recursive's Twin ‚Äî the default since Git 2.33. Rewrites the old Recursive strategy with better handling of criss-cross merges and rename conflicts. Faster, safer, same results.

$git merge -s ort feature # explicit (default anyway)
‚úì Modern Git default ‚úì Best for typical merges
üîÑ
Recursive Strategy
git merge -s recursive feature
üìö Analogy: The original postal sorting system ‚Äî reliable, proven, but slower on complex packages. Replaced by ORT but still works perfectly for most cases.

Was the default before Git 2.33. Handles criss-cross merge scenarios by recursively merging multiple common ancestors. Still available and still used in scripts/CI that pin behavior. The ORT strategy supersedes it.

$git merge -s recursive feature
‚Üí Legacy / older Git versions ‚Üí Behavior compatibility
‚ö°
Resolve Strategy
git merge -s resolve feature
üìö Analogy: A quick, no-frills mediator that only handles simple disputes ‚Äî two parties, one common reference point. If the conflict is complex, it gives up rather than guessing.

A simpler, older three-way merge strategy. Only works with exactly two heads. Faster than recursive for simple cases but can't handle complex merge histories. Rarely used today ‚Äî mostly historical.

$git merge -s resolve feature
‚Üí Simple 2-head merges ‚úó Fails on complex histories
üé≠
Ours Strategy
git merge -s ours feature
üìö Analogy: You hold a formal vote but already decided the outcome. The other branch's code is completely ignored ‚Äî but the merge commit exists for record-keeping purposes.

Discards all changes from the other branch entirely, keeping only your working tree. Creates a merge commit so history records that the merge happened, but the result is identical to your branch's current state. Note: -s ours ‚â† -X ours.

$git merge -s ours abandoned-feature
# "We're aware it existed, but we don't want its code"
‚Üí Retire old branches ‚úó Other branch code is lost ‚úì Tracks merge in history
üå≥
Subtree Strategy
git merge -s subtree other-repo
üìö Analogy: You're merging a supplier's entire warehouse into one corner of your bigger warehouse. The supplier's layout maps to your /vendor/lib subdirectory.

Like recursive, but Git auto-detects or you specify that one tree should be treated as a subtree of the other. Used to embed one project inside another. An alternative to git submodules for vendored dependencies.

$git merge -s subtree --prefix=vendor/ lib-repo/main
‚Üí Vendoring dependencies ‚úì Self-contained repo
Strategy Modifiers

Pass with -X flag to tune how conflicts are resolved during a merge. These work with ORT and Recursive strategies.

-X ours
Auto-favor our side in every conflict. Instead of stopping and asking you, Git picks the current branch's version. Unlike -s ours, the other branch's non-conflicting changes still come in.
-X theirs
Auto-favor their side in every conflict. The incoming branch wins all disputes. Useful when you know the feature branch is "more correct" or you're pulling in an authoritative update.
-X patience
Uses a smarter diff algorithm (diff --patience) that produces more human-readable diffs and fewer spurious conflicts. Slower but better for complex code with many repeated lines (e.g., lots of braces).
-X ignore-space-change
Ignores changes in whitespace count when detecting conflicts (tabs‚Üíspaces, extra spaces). Code changes still matter; only whitespace-only changes are ignored.
-X ignore-all-space
Ignores all whitespace entirely when comparing lines. Stronger than ignore-space-change. Useful after bulk formatting runs.
--allow-unrelated-histories
Required when merging two repos that share no common ancestor commit. Git normally refuses this. Example: merging a freshly initialized local repo with a remote that has its own history.
# Real-world examples:
$git merge -X ours feature # we always win conflicts
$git merge -X theirs feature # feature always wins conflicts
$git merge -X patience feature # smarter diff
$git merge -X ignore-space-change feature # ignore whitespace
$git merge --allow-unrelated-histories origin/main # different repos

‚ö†Ô∏è CRITICAL DISTINCTION: -s ours vs -X ours

-s ours (strategy)
Their entire branch is completely discarded. Only your files remain. Used to "absorb" a dead branch without its changes.
-X ours (modifier)
Their non-conflicting changes still come in. Only conflicts auto-resolve to our version. Much more useful day-to-day.
Merge Management Commands

These commands let you control the merge process when it's in progress ‚Äî especially when you hit conflicts.

üõë
git merge --abort
Bail out completely
üìö Analogy: You started mixing two paint colors and realized they clash badly. You throw the mixture out and restore the original cans exactly as they were.

Cancels a merge-in-progress and restores your working tree to the state before you ran git merge. All your edits to conflicted files are discarded. The branch pointer doesn't move.

$git merge feature # CONFLICT!
# Decided this isn't the right time
$git merge --abort
# Back to pre-merge state, clean slate
‚Üí Too many conflicts ‚Üí Wrong branch merged ‚Üí Need to rethink approach
‚ñ∂Ô∏è
git merge --continue
After resolving conflicts
üìö Analogy: You paused building IKEA furniture to find a missing part. Once you've found and placed it, you press "continue" on the assembly process.

After manually resolving conflicts and staging the resolved files with git add, this command completes the merge and creates the merge commit.

$git merge feature # CONFLICT!
# Edit conflicted files manually
$git add resolved-file.js
$git merge --continue # or: git commit
‚Üí After fixing conflicts ‚Üí Equivalent to git commit here
üö™
git merge --quit
Abandon without restoring
üìö Analogy: You stop assembling furniture mid-way and walk away, leaving the half-built chair on the floor. Git forgets the merge was happening, but your mess stays.

Like --abort but does NOT restore your files. Clears the merge state (MERGE_HEAD, etc.) while leaving your working directory as-is. Useful if you want to keep manual edits but start fresh with the merge tracking.

$git merge --quit
# Merge state cleared, files unchanged
‚Üí Rare, advanced use ‚Üí Keep edits, reset merge state
üîè
Signed Merge (GPG)
git merge --gpg-sign feature
üìö Analogy: Like notarizing a document. You cryptographically sign the merge commit with your private key, so anyone can verify it was actually you who merged this ‚Äî not a hacker.

Cryptographically signs the merge commit with your GPG key. Useful in security-sensitive projects where you need to verify the identity of who performed a merge. Requires GPG setup.

$git merge --gpg-sign feature
git merge -S feature # shorthand
$git log --show-signature # verify signatures
‚Üí Security-critical projects ‚Üí Open source maintainers
History-Rewriting Alternatives

Not true merges, but related workflows that achieve similar integration goals with different trade-offs on history.

ü™Ñ
Rebase
git rebase main
üìö Analogy: Instead of merging your edited draft with the original at the end, you secretly reprint all your notes as if you had started from the latest version of the original. The story is the same but the timeline looks cleaner.

Moves your branch's commits to a new base commit. Rewrites commit history ‚Äî commits get new SHAs. Creates a linear history. Never rebase commits that are already pushed to a shared branch.

$git checkout feature
$git rebase main
‚úì Linear clean history ‚úó Rewrites commits (new SHAs) ‚úó Never on shared branches
üéõÔ∏è
Interactive Rebase
git rebase -i HEAD~5
üìö Analogy: A film editor with the raw footage. You can reorder scenes, cut out bad takes, combine multiple shots into one, or rewrite dialog before the final cut is published.

Opens an editor showing the last N commits. You can reorder, squash, drop, edit, or reword any commit. The most powerful history-cleanup tool. Use before pushing to clean up a messy local branch.

# Opens editor with list like:
pick a1b2c3 Fix typo
squash d4e5f6 WIP: half-done feature
reword 7g8h9i Finish feature
drop bad000 "debugging remove me"
$git rebase -i HEAD~4
‚úì Polish before PR review ‚Üí Combine WIP commits ‚úó Local branches only
üèõÔ∏è
Rebase with Merge Preservation
git rebase --rebase-merges main
üìö Analogy: Moving a complex filing system to a new office. Instead of flattening all folders into one pile, you move everything while preserving the entire folder structure ‚Äî just with the new address.

By default, rebase flattens merge commits. --rebase-merges preserves the merge commit topology while still rebasing onto the new base. Essential when rebasing a branch that has its own internal merges.

$git rebase --rebase-merges main
# Keeps internal merge structure intact
‚Üí Branches with internal merges ‚úì Preserves topology
üçí
Cherry-Pick
git cherry-pick a1b2c3
üìö Analogy: From a whole fruit basket (another branch), you reach in and grab exactly the one cherry you want ‚Äî without taking the basket. One specific commit, applied onto your current branch.

Applies a specific commit from any branch onto your current branch. Creates a new commit with the same changes but a new SHA. Great for hotfixes ‚Äî apply a fix to both main and an older release branch.

$git cherry-pick a1b2c3 # one commit
$git cherry-pick a1b2c3..d4e5f6 # range
‚Üí Hotfix to multiple branches ‚Üí Cherry-pick only needed commits ‚ö† Creates duplicate commits
‚Ü©Ô∏è
Revert Merge Commit
git revert -m 1 HEAD
üìö Analogy: Instead of tearing up the signed contract (rewriting history), you issue a formal amendment that says "we're undoing everything in that last agreement." The original contract still exists in the records.

Creates a new commit that undoes the changes introduced by a merge commit. Safe for shared branches ‚Äî doesn't rewrite history. The -m 1 flag tells Git which parent is the "mainline" (usually 1 = your branch).

$git revert -m 1 HEAD
# -m 1 = keep the mainline (first parent = your branch)
# -m 2 = keep the merged branch's content
$git push # safe to push!
‚úì Safe for shared branches ‚úì History preserved ‚ö† Re-merging same branch needs care

üèÜ The Golden Rules
üìå Never rebase shared/public branches. Rebase rewrites history ‚Äî others who pulled the old commits will have a bad time.
üìå Use --no-ff for feature branches on team projects so you can see clearly where features begin and end in the log.
üìå -s ours ‚â† -X ours. Strategy discards all; modifier just auto-resolves conflicts in your favor.
üìå Prefer revert over reset when undoing published merges. Revert creates a new commit ‚Äî history is preserved.
üìå Squash for clean PR merges when individual commits are WIP notes. Use 3-way for preserving meaningful history.
üìå Always git merge --abort if you're unsure how to resolve a conflict ‚Äî there's no shame in starting fresh.
